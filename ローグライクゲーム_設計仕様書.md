# ローグライクゲーム 設計仕様書（ドラフト）

> 目的：本ドキュメントは「面白いローグライク／ローグライト」を成立させるための設計指針と仕様をまとめたもの。
> 具体的なテーマ（ファンタジー／SF／カードバトル等）は差し替え可能な“テンプレート”として記述する。

---

## 1. ゲーム概要

| 項目 | 内容 |
|------|------|
| ジャンル | ローグライト（プロシージャル生成 ＋ パーマデス ＋ メタプログレッション） |
| 視点 | 2D見下ろし型 もしくは サイドビュー（実装難易度低）／ 純ターン制グリッドも可 |
| 1プレイ時間 | 1ラン 20〜45分（"あと1回"を成立させる長さ） |
| コアの快感 | 「毎回違う状況で、自分だけのビルドを組み、際どい判断で生き延びる」 |
| プラットフォーム | PC（Steam）想定。後にコンソール／モバイル展開を考慮 |
| ターゲット | 自分で工夫・最適化するのが好きな層。難易度より“理解の深まり”で達成感を得る層 |

---

## 2. デザインの3本柱（Design Pillars）

設計判断に迷ったら、この3本柱に立ち返って意思決定する。

1. **意味のある選択（Meaningful Choices）**
   すべての選択肢にトレードオフを持たせる。「とりあえず強い1択」を作らない。
2. **公平なランダム性（Fair Randomness）**
   ランダム性は“新鮮さ”のため、公平性は“意思決定の納得感”のため。運だけで死なせない。
3. **知識が報われる（Knowledge is Power）**
   キャラは死んでもプレイヤーの理解は蓄積する。理解の深さが攻略を進める主軸。

---

## 3. ゲームループ

### 3.1 マクロループ（メタ進行：ラン間）
```
ラン開始 → 探索・戦闘・ビルド構築 → 死亡 or クリア
        → 報酬（メタ通貨・解放要素）獲得
        → 拠点で恒久強化／新要素アンロック
        → 次ランへ（より強い状態 or 新しい遊び方で再挑戦）
```

### 3.2 ミクロループ（ラン内：数十秒〜数分）
```
部屋に入る → 状況把握（敵・地形・宝箱を読む）
          → リスク判断（戦う/回避/アイテム使用）
          → 戦闘・解決 → 報酬選択（ビルドの分岐点）
          → 次の部屋へ
```

**設計原則**：ミクロループの1周ごとに「小さな選択」を、フロア／ステージ単位で「大きな選択」を必ず挟む。プレイヤーが“ただ進むだけ”の時間を作らない。

---

## 4. コアメカニクス

### 4.1 プロシージャル生成
- **生成対象**：マップ構造（部屋・通路）、敵配置、アイテム・宝箱、イベント、ボス前の構成。
- **生成方針**：完全ランダムではなく**制約付き生成（constrained generation）**。
  - 例：部屋数・難易度カーブ・必須要素（出口、ショップ、回復）は保証する。
  - 「クリア不能なシード」を作らないバリデーションを生成後に実行。
- **シード（seed）システム**：同一シードで同一マップを再現可能に。
  - デバッグ／競技（同条件タイムアタック）／バグ報告に有用。
- アルゴリズム候補：BSP分割、ドランカーズウォーク、ルームテンプレート＋接続グラフ、WFC（Wave Function Collapse）など。実装コストと表現したい構造で選ぶ。

### 4.2 パーマデス
- 死亡＝そのランの進行を全喪失。ただしメタ通貨・アンロックは残す（ローグライト型）。
- **死の納得感**が最重要。死因がプレイヤーに明確に伝わること（「何にやられたか」「回避可能だったか」）。
- 後半に“復活アイテム”を限定的に配置すると緊張と救済のバランスが取れる（使いどころが新たな判断になる）。

### 4.3 ターン構造 / アクション
- **ターン制グリッド型**：1手の重みが最大化。じっくり考える層向け。実装は比較的軽い。
- **リアルタイムアクション型**：プレイヤースキル（操作）が介在。間口は広いが実装・調整コスト大。
- どちらでも、**「時間をかけて状況を読む余地」**を残すこと（アクションでも一時停止やテレグラフ＝予兆表示で対応）。

---

## 5. ビルドシステム（面白さの主戦場）

ローグライクの満足度は、ここの“深さ”でほぼ決まる。

### 5.1 構成要素
- **アクティブ要素**：武器、スキル、消費アイテム（プレイヤーが能動的に使う）。
- **パッシブ要素**：レリック／遺物、ステータス補正、常時発動効果。
- **シナジー（synergy）**：要素同士の掛け算で1+1=3になる組み合わせ。
  - 例：「攻撃時に毒付与」×「毒の敵に追加ダメージ」など。

### 5.2 ビルド設計の原則
- **多様性**：勝ち筋（アーキタイプ）を最低3〜5系統用意（火力型／状態異常型／防御反射型／召喚型 等）。
- **発見性**：強力なシナジーは説明しすぎない。プレイヤーが“気づく”喜びを残す。
- **天井と床**：尖ったビルドほど上限は高く、下限（事故率）も高い、というリスク設計にする。
- **罠の回避**：完全な死にビルド（何を取っても勝てない）を作らない。最低限の保証ラインを敷く。

### 5.3 報酬選択の設計
- 報酬は**「3択提示」**を基本とし、毎回トレードオフを内包させる。
  - 例：高火力だがHP減 / 安定だが地味 / ギャンブル性が高いが化ける。
- ショップ／祭壇など**「リソースと引き換えに能動的に選ぶ」**仕組みを併設。

---

## 6. プログレッション設計

### 6.1 ラン内成長（一時的）
- レベルアップ、アイテム獲得、ビルド強化。死亡で全消失。
- 1ランの中で「弱い→強い」のパワーカーブを体感させる（中盤で“化ける”瞬間を設計）。

### 6.2 メタ進行（恒久）
- **メタ通貨**でキャラ恒久強化・新要素アンロック。
- **設計上の最重要バランス**：
  - グラインド過多 ✕（“作業”でしか進まないと飽きる）
  - 強くなりすぎ ✕（メタ強化だけで勝てるとランの緊張が消える）
  - **理想**：メタ進行は“選択肢の拡張”が中心。純粋な数値強化は控えめに。
- アンロックは**「新しい遊び方」**を解放するものを優先（新キャラ、新武器系統、新ルール）。

### 6.3 知識プログレッション（最重要・暗黙の進行）
- 数値で表れない「プレイヤーの理解」こそ主成長軸。
- これを促すため、**アイテム効果・敵パターンを一貫させ、学習可能にする**こと。

---

## 7. 難易度設計

- **動的難易度ではなく、明示的な難易度層**を推奨（攻略の達成感が明確になる）。
  - 例：クリア後に解放される「上昇難易度（昇格システム）」。Slay the Spireのアセンション、Hadesの熱量(Heat)型。
- **難易度カーブ**：序盤は失敗から学べる猶予を、終盤は知識とビルドの総合力を問う。
- **理不尽の排除**：
  - 初見殺しは“予兆（テレグラフ）”で回避可能にする。
  - 連続事故（リカバリ不能な負の連鎖）を生成段階で抑制。

## 8. コンテンツ要素（ボリューム指針）

| 要素 | 目安・指針 |
|------|-----------|
| 敵 | 行動パターンが識別できる程度に種類を確保。色違い水増しは避け、各敵に“対処法”を持たせる |
| ボス | フロア／ステージ区切りに配置。パターン学習型。初見の緊張と再戦の攻略快感を両立 |
| アイテム／レリック | シナジーを意識し“組み合わせ前提”で設計。単体最強を作らない |
| イベント部屋 | リスク選択（ギャンブル、取引、犠牲）を提供。物語の彩りも担う |
| ショップ | リソース管理の意思決定ポイント。リロール等の能動性を持たせる |

**設計原則**：コンテンツ量より“相互作用の数”。少ない要素でも掛け算が多ければ深くなる（Balatroが好例）。

## 9. UI / UX / フィードバック

- **可読性（readability）最優先**：今何が起きているか、なぜ死んだかが一目で分かること。
- **情報の透明性**：アイテム効果・確率・ダメージ計算は確認可能に（隠す快感より、判断材料の提供を優先）。
- **死亡画面**：到達度・ビルド・死因・統計を提示。「次への学び」と「成果の振り返り」を残す。
- **テンポ**：周回前提なので、リトライまでの摩擦（読み込み・演出）を最小化する。

---

## 10. リプレイ性を高める仕組み（チェックリスト）

- [ ] 毎ラン状況が有意に変わる（生成の多様性）
- [ ] 複数の勝ち筋（ビルドの分岐）が成立する
- [ ] 上昇難易度などの“長期目標”がある
- [ ] アンロックが“新しい遊び方”を開く
- [ ] デイリーチャレンジ／シード共有などの競技性（任意）
- [ ] 実績・コレクション要素（網羅欲を刺激）
- [ ] 1ランが短く、リトライ摩擦が低い

---

## 11. 技術的考慮（実装メモ）

- **シード管理**：乱数生成器（RNG）をシステム別に分離（マップ用／ドロップ用 等）。再現性とデバッグ性を確保。
- **生成バリデーション**：生成直後に「到達可能性チェック（経路探索）」を必ず走らせる。
- **データ駆動設計**：敵・アイテム・イベントを外部データ（JSON/設定ファイル）化し、バランス調整をコード変更なしで回せるように。
- **テレメトリ**：プレイデータを収集し、バランス調整の根拠にする（後述KPI）。

## 12. 評価指標（KPI）— “面白さ”を数値で観測する

体感の「面白い」をデータで裏取りするための指標。

| 指標 | 見るもの | 危険信号の例 |
|------|---------|-------------|
| ラン完走率（クリア率） | 難易度の妥当性 | 低すぎ＝理不尽 / 高すぎ＝緊張不足 |
| 平均ラン時間 | ペーシング | 想定と乖離＝間延び or 短すぎ |
| リトライ率（死亡→即再挑戦） | “あと1回”の強さ | 低い＝挫折感が強い |
| ビルド分布 | 多様性 | 1ビルドに偏る＝バランス崩壊（罠/最適解の存在） |
| 死因分布 | 公平性 | 特定要因に死が集中＝理不尽な可能性 |
| アイテム採用率／勝率 | 個別バランス | 採用0%＝死にアイテム / 採用100%＝壊れ |
| 7日／30日継続率（リテンション） | 長期の引力 | 低下＝飽きの早さ |

> ※採用率と勝率は**分けて**見る。「よく取られるが勝てない（罠アイテム）」「あまり取られないが勝てる（過小評価）」を区別するため。
> ※小さな勝率差を“調整すべき差”と断じる前に、サンプル数と統計的有意性を確認すること（過剰調整＝過学習を避ける）。

## 13. 最小実装（MVP）スコープ案

検証を最速で回すための最小構成。

1. 1キャラ ＋ 武器3種
2. 自動生成マップ（1ステージ＋ボス1体）
3. アイテム／レリック 15〜20種（シナジーが最低3系統成立する選定）
4. パーマデス ＋ メタ通貨1種 ＋ アンロック数個
5. 死亡画面と基本統計（死因・到達度）

→ この時点で**「あと1回やりたくなるか」**だけを検証する。面白さの核（ループ＋ビルド＋公平なランダム性）が機能しているかを最優先で確かめ、コンテンツ量やテーマ性の作り込みは後回しにする。

### 13.1 MVPで“検証する仮説”を一文で固定する
作り始める前に、検証したい仮説を1文に落とす。例：

> 「制約付き生成で公平なランダム性を担保し、3系統のシナジーが成立するビルドを与えれば、プレイヤーは死んでも“あと1回”やりたくなる」

この仮説に対し、第12章のKPI（特にリトライ率・ビルド分布）で Yes / No を判定する。仮説が外れたら、コンテンツを足す前に**ループかビルドのどちらが弱いか**を切り分ける。

### 13.2 MVPで“あえて作らないもの”（スコープ外）
最小検証を濁らせないため、初期は以下を意図的に除外する。

- 複数ステージ／複数ボス（まず1ステージで“核”を見る）
- 凝った演出・カットシーン・ストーリー（テンポ計測の邪魔になる）
- 2人目以降のキャラや上昇難易度（多様性は核の確認後）
- グラフィックの作り込み（プレースホルダで可。面白さは仕様で決まる）

### 13.3 MVPの“完成”判定ライン（Definition of Done）
- [ ] 1ランを最初から最後（クリア or 死亡）まで遊べる
- [ ] 生成バリデーション（到達可能性チェック）が常に通る
- [ ] 3系統のビルドが「実際に成立して勝てる」ことを開発者が確認済み
- [ ] 死亡画面で死因・到達度・ビルドが振り返れる
- [ ] 主要KPI（リトライ率・平均ラン時間・ビルド分布）を計測できる

## 14. 開発ロードマップ（MVP後の拡張順）

MVPで“核”が成立したと判定できたら、第2章の3本柱を崩さない順序で広げる。

| フェーズ | 主目的 | 主な追加要素 |
|---------|--------|------------|
| Phase 1（MVP） | 面白さの核の検証 | 1キャラ・武器3・1ステージ＋ボス・アイテム15〜20・メタ通貨1 |
| Phase 2（拡張） | 多様性とリプレイ性 | ステージ／ボス追加、アーキタイプを5系統へ、イベント部屋、ショップのリロール |
| Phase 3（長期目標） | 飽きの先延ばし | 上昇難易度（昇格システム）、2人目以降のキャラ、新ルール解放 |
| Phase 4（外向き） | コミュニティと競技性 | シード共有、デイリーチャレンジ、実績・コレクション、ローカライズ |

**順序の原則**：数値強化より“選択肢の拡張”を、コンテンツ量より“相互作用の数”を先に増やす（第6章・第8章の方針と一致）。各フェーズ後に必ずKPIで効果を裏取りし、外れた施策は足し続けず巻き戻す。

## 15. 設計判断チェックリスト（迷ったときの自己点検）

新しい要素を入れる前に、3本柱に照らして自問する。

- [ ] **意味のある選択**：この選択肢にトレードオフはあるか？ 「とりあえず強い1択」になっていないか？
- [ ] **公平なランダム性**：この運要素で“何もできずに死ぬ”ことはないか？ 予兆（テレグラフ）で回避可能か？
- [ ] **知識が報われる**：効果やパターンは一貫していて、プレイヤーが学習できるか？
- [ ] この要素は“相互作用（シナジー）”を増やすか、それとも単なる水増しか？
- [ ] 死因・効果・確率はプレイヤーから見て透明か？
- [ ] 入れたあと、KPIのどの指標で良し悪しを判定するか決まっているか？
