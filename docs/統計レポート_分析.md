# KPI統計レポートと分析（データ駆動バランス調整の記録）

設計仕様書 §12「“面白さ”を数値で観測する」に従い、Bot に **1,000ラン** 自動プレイさせて
KPI を計測し、その数値を根拠にバランスを調整した記録。
生の出力は [`KPIレポート_1000runs.txt`](./KPIレポート_1000runs.txt)。再生成は `python3 -m roguelike sim -n 1000`。

> 注意（§12の但し書き）：これは「平均的なヒューリスティックBot」の値。人間プレイヤーの
> 体感とは差が出る。小さな勝率差を過剰に追い込む（＝Botへの過学習）ことは避け、
> 「分布が健全か」を主眼に判断した。

## 1. 最終的な数値（1,000ラン）

| KPI | 値 | 判定 |
|-----|----|------|
| クリア率 | **58.5%** | 手応えのある帯域（理不尽でも緊張不足でもない） |
| 平均ラン時間 | 288ターン（中央278） | 短く周回でき、最長1104で**停滞ゼロ** |
| ビルド分布（クリア） | thorns31 / power28 / poison21 / sustain16 / utility4 % | **5系統すべてが勝てる**＝多様性◎ |
| 武器別勝率 | 短剣52.7 / 剣59.3 / 大槌64.0 % | 健全な範囲に収束 |
| 死因 | ボス63 / 射手15 / 大兵14 / その他8 % | トラッシュmob偏重なし |

到達フロア：F5（ボス階）到達 86%、F3で詰む 3% → 序盤は学習の猶予、終盤に総合力を問う（§7）。

## 2. 調整の過程（数値→診断→施策）

「作って測って直す」を3周回した。各周の数値が施策の根拠になっている。

### 周回1：初期実装
- **計測**：クリア率 **0%**、さらに約30%のランがターン上限に張り付いて終了しない。
- **診断**：(a) Botが逃げ回る治癒師を延々追って停滞、(b) 難易度が理不尽。
- **施策**：Botの主目標を「出口」に変更＋アンチストール（長居でrushモード）。

### 周回2：停滞の根治と難易度
- **計測**：クリア率 7%、死因の **60%が射手** に集中。武器勝率が 短剣3% / 大槌19% と崩壊。
  毒ビルドの勝率がほぼ 0%。
- **診断**：
  - 射手が高火力・長射程で理不尽（§7違反）。
  - **バグ発見**：短剣の `bonus_poison` が「毒の刃」レリック内でしか発火せず、武器固有の
    毒が常時乗らない。毒ビルドが特定レリック依存の“死にビルド化”していた（§5.2違反）。
  - 攻撃力不足で戦闘が長引き、結果として被弾が増え攻撃偏重が最適化されていた。
- **施策**：射手を攻撃5→3・射程8→5に。武器固有の毒を engine 側で常時加算（毒に床を作る）。
  踏破時に最大HP22%回復（連続事故の緩和 §7）。プレイヤーHP 40→46。敵密度を緩和。

### 周回3：仕上げと退化生成の排除
- **計測**：クリア率 66%。だが約20%のランがまだ停滞。原因を追うと
  **治癒師が偶然6体出現**してマップを封鎖し、Botが2マスを686回往復していた。
- **診断**：特定の敵が群れると「対処不能で退屈」な退化局面が生まれる（§4.1の制約不足）。
- **施策**：
  - **制約付き生成**：治癒師は1フロア最大1体、大兵は最大2体に上限（degenerate構成を排除）。
  - **Bot rush の堅牢化**：敵を障害物とみなさず出口最短路を進み、道中の敵は殴り倒す
    → 連結マップなら必ず終了し、振動も起きない（停滞 **0%** を達成）。
  - 大槌を防御-3に、ボスHPを66に、回復22%に微調整 → クリア率 58.5% に着地。

## 3. このレポートが体現していること

- **死因が単一トラッシュmobに集中していない**：かつて射手60%だったのを生成・数値の両面で是正。
  現在ボスに集中するのは「最終ボスで力尽きる」＝設計意図通り（クライマックス）で、理不尽ではない。
- **採用率と勝率を分離**（§12但し書き）：例「鉄壁」は採用時勝率88%だが、これは“勝てる盤面で
  取られやすい”相関でもあるため、単独で壊れと断ぜず分布全体で評価した。
- **罠アイテムの不在**：採用率0%・突出した死にアイテムは無く、最低保証ライン（§5.2）を満たす。

## 4. どれくらいのデータが必要だったか（運用の指針）
- **n=40〜60**：方向性（停滞の有無、致命的偏り）の即把握には十分。開発中の高速イテレーション用。
- **n=300〜500**：ビルド分布・武器勝率など「分布の形」を信頼して読める。調整の意思決定はこの規模で実施。
- **n=1,000**：最終確認・レポート用。これ以上は小数点以下の揺れで、過学習リスクの方が大きい。

→ 本MVPの結論：**“面白さの核（ループ＋ビルド＋公平なランダム性）は成立している”**。
   次の拡張（ステージ追加・昇格難易度・新キャラ）は §14 ロードマップの順に。

## 5. リバランス更新 v0.2（難易度↑＋シナジー強化＋コンボ計測）

**なぜ**：手応えをもう一段引き上げ（目標クリア率≈40%）、かつ「単体最強」を避けて
「組み合わせ（シナジー）が報われる」状態にするため。あわせて**特定アイテム組み合わせ別の
クリア率**を計測する KPI を追加し、シナジーの効きをデータで確認できるようにした。
全変更は `data.json`（数値真値）で行い、Python sim で再検証 → TS は同一 data.json を読むため
等価性（bit一致）を自動的に維持（ゴールデン再生成で確認済み）。

### 5.1 最終 KPI（n=1000, seed=1000）
| 指標 | v0.1（旧凍結） | v0.2（本更新） | 判定 |
|---|---|---|---|
| クリア率 | 58.5% | **40.6%** | 目標≈40%、健全帯(35〜65%)内 |
| 死因（ボス） | 62.9% | 66.8% | クライマックス集中・トラッシュ偏重なし |
| クリア時ビルド分布 | thorns31/power28/poison21/sustain16/utility4 | thorns31/power29/poison20/sustain17/utility3 | **5系統すべて勝利**・持続の突出を解消 |
| 停滞 | 0% | 0% | 維持 |

### 5.2 変更点（data.json）と狙い
- **難易度↑**：ボスHP 66→78、踏破回復 0.22→0.16、プレイヤーHP 46→44。
  → 死因をボス中心に保ったままクリア率を 58.5%→40.6% に。
- **単体最強の是正**：鉄壁 def+4/HP+10 → def+3/HP+6 → **def+2/HP+4**（鉄壁を含むペアが軒並み88〜100%だった偏りを緩和）。
  持続も吸血30%→24%・活力+25→+20 に調整（持続コンボの突出 +31pt→+22.7pt）。
  - **v0.2.1 追加是正（因果A/B駆動）**：複数seedの `npm run ab`（採用時勝率の交絡を排した因果Δ）で鉄壁が **+43pt と全17レリック中ダントツ1位**＝「とりあえず強い1択」と確証。def+3/HP+6 → **def+2/HP+4** に弱体。結果、因果Δ +43pt→**+34pt**（再生/吸血/守りの心得と同列）、採用時勝率 72.2%→**51.6%**、クリア率 44.3%→**40.3%**（健全帯維持・5系統勝利・ASSERT OK）。data.json のみ編集＋回帰テスト期待値とゴールデンを更新。
- **シナジー強化（噛み合うほど強く）**：
  毒＝毒の刃 付与3→5・腐食 +50%→+100%・疫病 伝播3→4（毒コンボが −29pt → **+7〜9pt** に転換）。
  火力＝狂戦士 最大+10→+13・双牙 30%→36%・巨人 HP-12→-10。反射＝棘の外殻 3→4。

### 5.3 コンボ別クリア率（新 KPI・抜粋）
`sim` レポートに「シナジーコンボ別クリア率」「相性の良いレリック2枚組（共起トップ）」を追加。
基準40.6%に対し：**持続(吸血×再生) +22.7pt / 毒(付与×腐食) +9.4pt / 毒(付与×増幅) +7.9pt /
反射(外殻×報復) +4.4pt** が正（＝噛み合うほど報われる）。一方 **反射(鎧×外殻) −16pt** や
**火力(狂×連撃) −6.4pt** は弱く、今後の調整候補として可視化された（採用率と勝率を分けて評価：§12）。

> 注：本更新で凍結基準は v0.2 に更新。今後の数値変更も「data.json 変更 → Python sim 再検証 →
> 健全帯確認 → 本レポート/KPI/設計仕様書 同時更新」（開発運用ガイド）に従う。

## 6. v0.3 更新（踏破回復の2段階化 ＋ 2体目ボス追加）

**なぜ**：回復の起伏とボス戦の多様性を増やすため。(1) 踏破回復を「通常／たまに休憩(大回復)」の2段階に、
(2) ボス階を2種からランダム出現に。検証・記録は JS 一本化後のため **Node 版 sim**（`web/scripts/sim.ts`）で実施。

### 6.1 変更（`web/data.json`）
- `run`：`restHeal` 0.34・`restChance` 0.25 を追加（通常 `floorClearHeal` 0.16 はそのまま）。`bossPool: ["boss","twin_beast"]`。
- 敵に **`twin_beast`「双牙の獣」**（hp70/atk9/def2・behavior `boss_twin`）を追加。深淵の王（AoEスラム＋増援＝距離管理型）と対になる**近接連撃型**（隣接で攻撃＋35%追撃、AoE/召喚なし）。

### 6.2 最終 KPI（Node sim, n=1000, seed=1000）
| 指標 | v0.2 | v0.3 | 判定 |
|---|---|---|---|
| クリア率 | 40.6% | **44.3%** | 健全帯(35〜65%)・目標≈40%帯 |
| 死因（ボス） | 深淵66.8% | **深淵34.8% / 双牙33.8%** | 2ボスがほぼ拮抗＝偏りなく両方が脅威 |
| クリア時ビルド | 5系統 | **5系統すべて勝利** | 多様性維持 |
| 停滞 | 0% | 0% | 維持（下記の調整で確保） |

### 6.3 調整の過程
- 初期値（restChance0.33/restHeal0.40・twin hp60/atk8）→ クリア率 **51%**（休憩で易化・双牙が弱め）。
- 休憩を控えめに（0.25/0.34）＋双牙を強化（hp70/atk9）→ **44%**、深淵≒双牙の死因バランスに。
- **停滞の根治**：双牙が猛追し続けるためボス階で Bot が追走ループ（n=200で3件停滞）。Bot のアンチストールを
  ボス階でも有効化し「長居したらボスへ直行」に修正（ゲーム本体ではなく計測Botの探索修正）→ 停滞0%。
- gen-fixtures でゴールデン再生成、`npm run check`（doctor＋test＋sim assert＋build）緑。
