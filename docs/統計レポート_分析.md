# KPI統計レポートと分析（データ駆動バランス調整の記録）

設計仕様書 §12「“面白さ”を数値で観測する」に従い、Bot に **1,000ラン** 自動プレイさせて
KPI を計測し、その数値を根拠にバランスを調整した記録。
生の出力は [`KPIレポート_1000runs.txt`](./KPIレポート_1000runs.txt)。再生成は `python3 -m roguelike sim -n 1000`。

> 注意（§12の但し書き）：これは「平均的なヒューリスティックBot」の値。人間プレイヤーの
> 体感とは差が出る。小さな勝率差を過剰に追い込む（＝Botへの過学習）ことは避け、
> 「分布が健全か」を主眼に判断した。

---

## 1. 最終的な数値（1,000ラン）

| KPI | 値 | 判定 |
|-----|----|------|
| クリア率 | **58.5%** | 手応えのある帯域（理不尽でも緊張不足でもない） |
| 平均ラン時間 | 288ターン（中央278） | 短く周回でき、最長1104で**停滞ゼロ** |
| ビルド分布（クリア） | thorns31 / power28 / poison21 / sustain16 / utility4 % | **5系統すべてが勝てる**＝多様性◎ |
| 武器別勝率 | 短剣52.7 / 剣59.3 / 大槌64.0 % | 健全な範囲に収束 |
| 死因 | ボス63 / 射手15 / 大兵14 / その他8 % | トラッシュmob偏重なし |

到達フロア：F5（ボス階）到達 86%、F3で詰む 3% → 序盤は学習の猶予、終盤に総合力を問う（§7）。

## 2. 調整の過程（数値→診断→施策）

「作って測って直す」を3周回した。各周の数値が施策の根拠になっている。

### 周回1：初期実装
- **計測**：クリア率 **0%**、さらに約30%のランがターン上限に張り付いて終了しない。
- **診断**：(a) Botが逃げ回る治癒師を延々追って停滞、(b) 難易度が理不尽。
- **施策**：Botの主目標を「出口」に変更＋アンチストール（長居でrushモード）。

### 周回2：停滞の根治と難易度
- **計測**：クリア率 7%、死因の **60%が射手** に集中。武器勝率が 短剣3% / 大槌19% と崩壊。
  毒ビルドの勝率がほぼ 0%。
- **診断**：
  - 射手が高火力・長射程で理不尽（§7違反）。
  - **バグ発見**：短剣の `bonus_poison` が「毒の刃」レリック内でしか発火せず、武器固有の
    毒が常時乗らない。毒ビルドが特定レリック依存の“死にビルド化”していた（§5.2違反）。
  - 攻撃力不足で戦闘が長引き、結果として被弾が増え攻撃偏重が最適化されていた。
- **施策**：射手を攻撃5→3・射程8→5に。武器固有の毒を engine 側で常時加算（毒に床を作る）。
  踏破時に最大HP22%回復（連続事故の緩和 §7）。プレイヤーHP 40→46。敵密度を緩和。

### 周回3：仕上げと退化生成の排除
- **計測**：クリア率 66%。だが約20%のランがまだ停滞。原因を追うと
  **治癒師が偶然6体出現**してマップを封鎖し、Botが2マスを686回往復していた。
- **診断**：特定の敵が群れると「対処不能で退屈」な退化局面が生まれる（§4.1の制約不足）。
- **施策**：
  - **制約付き生成**：治癒師は1フロア最大1体、大兵は最大2体に上限（degenerate構成を排除）。
  - **Bot rush の堅牢化**：敵を障害物とみなさず出口最短路を進み、道中の敵は殴り倒す
    → 連結マップなら必ず終了し、振動も起きない（停滞 **0%** を達成）。
  - 大槌を防御-3に、ボスHPを66に、回復22%に微調整 → クリア率 58.5% に着地。

## 3. このレポートが体現していること

- **死因が単一トラッシュmobに集中していない**：かつて射手60%だったのを生成・数値の両面で是正。
  現在ボスに集中するのは「最終ボスで力尽きる」＝設計意図通り（クライマックス）で、理不尽ではない。
- **採用率と勝率を分離**（§12但し書き）：例「鉄壁」は採用時勝率88%だが、これは“勝てる盤面で
  取られやすい”相関でもあるため、単独で壊れと断ぜず分布全体で評価した。
- **罠アイテムの不在**：採用率0%・突出した死にアイテムは無く、最低保証ライン（§5.2）を満たす。

## 4. どれくらいのデータが必要だったか（運用の指針）
- **n=40〜60**：方向性（停滞の有無、致命的偏り）の即把握には十分。開発中の高速イテレーション用。
- **n=300〜500**：ビルド分布・武器勝率など「分布の形」を信頼して読める。調整の意思決定はこの規模で実施。
- **n=1,000**：最終確認・レポート用。これ以上は小数点以下の揺れで、過学習リスクの方が大きい。

→ 本MVPの結論：**“面白さの核（ループ＋ビルド＋公平なランダム性）は成立している”**。
   次の拡張（ステージ追加・昇格難易度・新キャラ）は §14 ロードマップの順に。
