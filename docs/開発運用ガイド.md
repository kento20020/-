# 開発運用ガイド

このリポジトリでゲーム開発を継続するためのブランチ運用とワークフロー。
**まずここを読む。** 仕様判断は `設計仕様書.md` の「3本柱」と「KPI」が最終基準。

---

## 1. ブランチモデル

| ブランチ | 役割 | ルール |
|---------|------|--------|
| `main` | 常に「遊べてテストが通る」安定版 | **直接 push 禁止。PR 経由でのみ更新** |
| 作業ブランチ | 1つの目的の作業 | `main` から分岐し、終わったら PR → マージ → 削除 |
| `claude/*` | Claude Code が自動生成する作業ブランチ | そのまま使ってよい（例：今の PR #1 のブランチ） |

### 作業ブランチの命名（人が切る場合）
`種別/短い説明`（英小文字・ハイフン区切り）。種別は次のいずれか：

- `feature/` 新要素（例 `feature/new-boss-phase`）
- `fix/` バグ修正（例 `fix/poison-tick`）
- `balance/` 数値・出現テーブル調整（例 `balance/archer-nerf`）
- `docs/` ドキュメントのみ
- `chore/` 雑務（CI・整理など）

> 現状：`main` にはまだゲーム以前の検討資料しか無く、ゲーム一式は PR #1 のブランチにある。
> **PR #1 をマージすると `main` がこのゲームリポジトリの正となる。** 以後は本ガイドの運用に従う。

## 2. 1サイクルの流れ

```bash
git switch main && git pull origin main          # 最新の安定版から
git switch -c feature/＜内容＞                     # 作業ブランチを切る
# …実装…
python3 -m roguelike.tests                        # 1) テスト
python3 -m roguelike sim -n 300                   # 2) KPIで影響を確認
git add -A && git commit -m "feat: ＜内容＞"       # コミット（規約は §3）
git push -u origin feature/＜内容＞                 # push
# → GitHub で Pull Request を作成 → レビュー → main にマージ → ブランチ削除
```

**原則：ミクロでもマクロでも「測ってから決める」。** 体感ではなく KPI を根拠にする。

---

## 3. コミットメッセージ規約

`種別: 要約`（日本語可）。本文に「なぜ」を書く。種別は §1 の作業ブランチと同じ語彙：
`feat / fix / balance / docs / refactor / test / chore`。

例：
```
balance: 射手を攻撃5→3・射程8→5に下げる

死因の60%が射手に集中していた（理不尽の兆候）。sim n=500 で死因分布が
ボス中心の健全な形に戻ることを確認。
```

---

## 4. マージ前チェックリスト（Definition of Done）

PR を `main` にマージする前に、すべて満たすこと：

- [ ] `python3 -m roguelike.tests` が緑（生成の到達性・再現性・シナジー・停滞なし）
- [ ] `python3 -m roguelike sim -n 300` 以上で KPI が健全
  - [ ] クリア率が手応えのある帯域（おおむね 35〜65%）
  - [ ] ビルド分布が複数系統に分かれている（1系統独占でない）
  - [ ] 死因が単一のトラッシュmobに偏っていない（ボス集中は可）
  - [ ] 停滞（ターン上限張り付き）が出ていない
- [ ] データを変えたら `docs/統計レポート_分析.md` と `KPIレポート_1000runs.txt` を更新
- [ ] 仕様に関わる変更は `docs/設計仕様書.md` に反映

## 5. コンテンツの追加方法（データ駆動）

ロジックを書き換えず、**`roguelike/data.py` への追記**で要素を増やせる。

- **敵**：`ENEMIES` に `EnemyType` を追加し、`SPAWN_TABLE` の tier に重みで登録。
  新しい挙動が要るときだけ `engine.py` の `_act_enemy` に behavior を1つ足す。
- **武器**：`WEAPONS` に `Weapon` を追加。
- **レリック**：`RELICS` に `Relic` を追加。効果はイベントフック
  （`on_attack` / `on_hit_taken` / `on_turn_start` / `on_kill`）で書く。
  既存フックの組み合わせで**シナジー**が自然に生まれる。

追加後は必ず `sim` で **採用率・採用時勝率・死因** を確認し、
死にアイテム（採用率ほぼ0）・壊れ（突出勝率）・理不尽（死因集中）を是正する。

---

## 6. バランス調整で測るデータ量の目安

| 規模 | 用途 | 所要（目安） |
|------|------|-------------|
| n=40〜60 | 方向確認・停滞や致命的偏りの即把握 | 数秒 |
| n=300〜500 | 分布の形を信頼して意思決定 | 約15〜25秒 |
| n=1000 | 最終確認・レポート用 | 約40秒 |

これ以上はばらつきが小さく、**Bot への過学習**のリスクが上回る（設計仕様書 §12 の但し書き）。

大きな節目では git タグを打つ（例 `git tag v0.1.0`）。

## 8. （推奨）main の保護と CI

GitHub の Settings → Branches で `main` に保護ルールを設定すると安全：

- Require a pull request before merging
- Require status checks to pass（下の CI を必須にする）

最小の CI 例（`.github/workflows/ci.yml`）：
```yaml
name: ci
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.11" }
      - run: python -m roguelike.tests
      - run: python -m roguelike sim -n 200
```
