# ローグライクゲーム 設計仕様書 — Web版実装仕様

> **本書の位置づけ**：「設計仕様書（§1〜§17）」の続編。§16・§17 で1,000ランのKPI検証を経て
> 凍結したPython版MVPを、プレイ可能なWebアプリとして公開するための確定仕様を記録する。
>
> 設計判断の最終基準は引き続き **§2「3本柱」** と **§12「KPI」**。
> 迷ったときは「意味のある選択か → 公平なランダム性か → 知識が報われるか → どのKPIで測るか」の順に自問する。

## §18. 何を変えて、何を変えないか

| 変えないもの（凍結） | 変えるもの・追加するもの |
|---|---|
| 5系統ビルド・レリック17種・敵7種の数値とバランス | 描画・操作・情報表示のレイヤー |
| 生成アルゴリズム・生成制約（治癒師≤1、大兵≤2） | エンジン言語（Python → TypeScript） |
| Botシム基盤とKPI計測（Python側で継続） | デプロイ先・データ管理方式 |
| 3本柱（§2）・KPI指標（§12） | データ外部化（data.py → data.json） |
| クリア率58.5%・停滞0%・5系統全勝のバランス | UIのパレット・テーマ（§23で後述） |

**「凍結」とは「変えてはいけない」ではなく、「変えたら必ずBotシムで再検証する」の意味。**
数値を変えたら n=300〜500 でKPIを確認し、健全帯（クリア率35〜65%、ビルド分布偏重なし、
死因トラッシュmob集中なし）に留まることを確かめてからmainにマージする（dev guide §4・§6）。

## §19. アーキテクチャ

### 19.1 全体構成

```
Python engine（roguelike/）
  ├── data.json ← 両エンジン共通の唯一の真値（§20）
  ├── engine.py（ゲームロジック・バランスの source of truth）
  └── Botシム（今後の調整・等価性検証はここで実施）

TypeScript engine（web/）
  ├── data.json（Python側と同じファイルを読む）
  ├── engine.ts（engine.py の忠実な移植）
  └── プレイ用フロント（Vite + DOM/CSS）
         ↓ 静的ビルド
    GitHub Pages（サーバ不要・即ロード）
```

### 19.2 技術選定の根拠

| 項目 | 選定 | 根拠 |
|---|---|---|
| エンジン言語 | TypeScript | 型安全・ブラウザネイティブ・バンドルが軽量 |
| 描画方式 | DOM/CSS グリッド + HTML オーバーレイ | §9「可読性・透明性」に必要なツールチップ・明確なテキスト・アクセシビリティはDOMが最適。ターン制グリッドは60fpsを必要としない |
| ビルドツール | Vite | 即時HMR・静的出力・TypeScript対応が軽量 |
| デプロイ | GitHub Pages | サーバ不要・シード再現性そのまま維持・無料 |
| Canvasを使わない理由 | — | 将来エフェクトが必要になったら「エフェクト専用Canvasレイヤー」を上に追加で対応できる。今は不要 |

### 19.3 二層運用ルール

- **バランス調整は必ずPythonで行う**。data.json を変更 → Python Botシムで検証 → 確認後にTS側へ反映。
- TS engine はゲームロジックを追加・変更しない。「UIとロジックの接続」のみを担う。
- 新しいレリック/敵を追加したい場合 → data.json に追記 → Pythonシムで検証 → 両エンジンに自動反映（dev guide §5の延長）。

## §20. データ設計（data.json 外部化）

### 20.1 方針

`roguelike/data.py` の ENEMIES / SPAWN_TABLE / WEAPONS / RELICS を `data.json` に外部化し、
PythonとTSが**同じファイル**を読む構成にする。これにより：

- バランス変更が一箇所で済み、二重管理が消える
- 移植するのはロジックだけ・一回だけ（数値は共有JSON）
- Pythonシムで検証した値がTS側にそのまま適用される

### 20.2 スキーマ概要

```jsonc
{
  "_meta": {
    "hookEvents": ["onAttack", "onHitTaken", "onTurnStart", "onKill"]
  },
  "run":    { "floors": 5, "bossFloor": 5, "floorClearHeal": 0.22 },
  "player": { "hp": 46, "atk": 2 },

  "weapons": [
    { "id": "dagger", "name": "短剣", "atk": 3,
      "hooks": [{ "trigger": "onAttack", "effect": "applyPoison", "amount": 1 }] },
    { "id": "sword",  "name": "剣",   "atk": 6, "hooks": [] },
    { "id": "maul",   "name": "大槌", "atk": 7, "statMods": { "def": -3 }, "hooks": [] }
  ],

  "relics": [
    // 17種。archetype / statMods / hooks で記述。詳細は data.json ドラフト参照
    { "id": "giants_blood", "name": "巨人の血", "archetype": "power",
      "statMods": { "atk": 6, "maxHp": -12 }, "hooks": [] }
    // …
  ],

  "enemies": [
    { "id": "archer", "name": "射手", "hp": "?", "atk": 3, "range": 5,
      "behavior": "lineShot", "telegraph": true, "spawnCap": null },
    { "id": "healer", "name": "治癒師", "behavior": "healAllies", "spawnCap": 1 },
    { "id": "brute",  "name": "大兵",  "behavior": "tanky",      "spawnCap": 2 }
    // 全7種。hp・atk は data.py から補完
  ],

  "spawnTable": {
    "1": [{ "id": "rat", "weight": null }, { "id": "slime", "weight": null }],
    "5": [{ "id": "boss", "weight": 1, "guaranteed": true }]
    // tier 2〜4 の重みは data.py から補完
  }
}
```

完全なドラフト（null埋め待ちの状態）は `data.json` に同梱。
**data.py の ENEMIES / SPAWN_TABLE / WEAPONS / RELICS を貼ることで即座に確定版が生成できる。**

### 20.3 フックイベント語彙

| trigger | 発火タイミング | 使用例 |
|---|---|---|
| `onAttack` | プレイヤーが攻撃したとき | 毒の刃（攻撃毎に毒付与） |
| `onHitTaken` | プレイヤーがダメージを受けたとき | 棘の鎧（反射ダメージ） |
| `onTurnStart` | ターン開始時 | 再生（HP回復）、狂戦士（HP減で攻撃増） |
| `onKill` | 敵を倒したとき | 疫病（毒伝播）、双牙（追撃） |
| `passive` | 常時発動（条件なし） | 深い傷（毒ダメ2倍）、腐食（毒敵へ+50%） |

## §21. UI/UXレイアウト

### 21.1 設計方針

**デスクトップ先行・情報密度優先。**
§9（可読性・透明性）で要求される情報を折りたたまず全面展開できることが、
デスクトップ先行を選んだ最大の理由。スマホ対応は将来フェーズで段組み崩し対応。

### 21.2 メインプレイ画面（3領域構成）

```
┌────────────────────────────────────────────────────┐
│  F2/5  ·  124手  ·  seed: 7F3A               [≡]  │  トップバー
├──────────────────────────┬─────────────────────────┤
│                          │  HP ████████░░  38/46  │
│                          │  攻 8   防 1   武器:剣  │
│    盤  面（グリッド）     ├─────────────────────────┤
│                          │  ビルド（状態異常型）    │
│  @ · · · a   #           │  [毒の刃][深い傷][腐食]  │
│  # . $ . .   #           │  [鋭利]                 │
│  # r . . >   #           │                         │
│                          │                         │
├──────────────────────────┴─────────────────────────┤
│  ▸ 射手があなたを狙っている（次ターン射撃）          │  ログ
│  ▸ スライムに6ダメージ / 毒+1付与                  │
│  ▸ F2に到達：最大HPの22%を回復                     │
└────────────────────────────────────────────────────┘
```

| 領域 | 役割 | §9との対応 |
|---|---|---|
| トップバー | フロア深度・ターン数・seed常設・メニュー | シード共有（§4.1）・テンポ把握 |
| 盤面（中央左） | グリッド・プレイヤー・敵・宝・予兆 | 可読性最優先・公平なランダム性（§7） |
| 右サイドバー | HP/ステータス + ビルド全展開（常時） | 情報の透明性（§9）・ビルド把握 |
| 下ログ | 戦闘履歴全文常駐 | 死の納得感（§4.2・§9） |

### 21.3 密度デスクトップで実現する追加情報層

| 機能 | 実装概要 | 3本柱 |
|---|---|---|
| **予兆の常時可視化** | 射手の射線・ボスAoEを色付きマスで常駐表示 | ②公平なランダム性 |
| **隣接時ダメージプレビュー** | 敵に隣接した瞬間「殴ると○ダメ／被弾△」を盤面近くに表示 | ①意味のある選択 |
| **ルックモード詳細パネル** | xキー起動→カーソル先の敵/タイル全情報を右パネルに展開 | ③知識が報われる |
| **ビルド全展開常駐** | レリックを畳まず常時表示。hover で効果詳細ツールチップ | ①+③ |
| **ログ全文常駐** | 折りたたまず戦闘履歴を遡れる | 死の納得感 |
| **seed常設表示** | 再現・競技性の基盤（§4.1・Phase4） | — |

### 21.4 情報透明性 vs 発見性（§5.2 vs §9）の解

段階的開示で両立する。

- **明示する**：単体の効果・数値・発動条件（ツールチップ）
- **説明しすぎない**：複数レリックの相乗効果（シナジー）はプレイヤーが気づく余地を残す

### 21.5 オーバーレイ（メイン画面上のモーダル）

| モーダル | 起動タイミング | 必須要素 |
|---|---|---|
| **報酬3択** | フロアクリア・宝箱 | カードUI・トレードオフ明示（例：攻+6 / 最大HP-12） |
| **死亡画面** | HP0 | 到達フロア・ビルド・死因・KPI統計。`R`で即リスタート・seed入力 |
| **ショップ** | Phase2実装 | リソース消費・リロール（§5.3） |
| **イベント部屋** | Phase2実装 | リスク選択テキスト（§8） |

## §22. 入力モデル

### 22.1 設計方針

**キーボード先行・4方向・WASD主操作。**

- WASD = 主操作。矢印キー = ミラー（どちらでも同じ動作）
- hjkl・テンキー2468 = 代替入力（受け付けるが画面には表示しない）
- マウス = 補助のみ（ホバーでツールチップ表示。移動・攻撃はキーボード主）

### 22.2 移動

| キー | 方向 |
|---|---|
| W / ↑ | 上 |
| A / ← | 左 |
| S / ↓ | 下 |
| D / → | 右 |
| Space / `.` / テンキー5 | 待機（1ターンパス） |

**4方向固定（斜めなし）。** エンジンの移動モデルに準拠。斜めを追加すると到達距離・被弾頻度・
死因分布が変わりKPI検証済みバランスが崩れるため、追加する場合はBotシム再検証が必須。

**bump-to-attack：** 敵のいるマスへの移動入力 = 攻撃。専用攻撃キー不要。

### 22.3 アクションキー

| キー | 機能 |
|---|---|
| `x` | ルックモード（カーソルで盤面を調べる・詳細+ダメージ予測） |
| `>` | 階段を降りる（出口マスで自動実行も可） |
| `o` | 自動探索（未踏へ自動歩行・予兆/新規敵で自動停止） |
| `Tab` / `i` | ビルド・所持品パネル |
| `Esc` | キャンセル / メニュー |
| `R`（死亡画面） | クイックリスタート（seed入力可） |

### 22.4 ルックモード（`x`）の仕様

ホバーのキーボード代替。§9「情報の透明性」と「死の納得感」を担う。

1. `x` を押すと盤面に半透明カーソルが現れる
2. WASD/矢印でカーソルを移動（通常移動と同じキー）
3. カーソル先の情報を右サイドバーに展開（敵のHP・行動・耐性・予兆状態）
4. 敵マスにカーソルが重なると「殴るとXダメ / 被弾Y」のダメージ予測を表示
5. `Esc` でルックモード終了・通常操作に戻る

### 22.5 意図的に除外したもの

| 機能 | 除外理由 |
|---|---|
| 自由休息（任意タイミングのHP回復） | 凍結バランスの根拠である「踏破時22%回復（§17）」を崩す |
| 斜め移動（8方向） | エンジンの移動モデル外。追加はバランス再検証が必要 |
| 自動攻撃（マウスクリックで移動+攻撃） | キーボード主の方針に反する。bump-to-attackで代替 |

## §23. テーマ・世界観（未確定）

**現状：差し替え可能な抽象ダンジョンのまま（§1・§16.1）。**

テーマは「ゲームの顔」であり、UIパレット・各系統の意味づけ・敵の名前・ボス名すべてに影響する。
決定後に data.json のテキストフィールドと CSS 変数（カラーパレット）を差し替えるだけで適用できる。

### 確定済みの候補

| テーマ | 降下の意味 | 毒系統の読み替え例 | パレット |
|---|---|---|---|
| 古典ファンタジー地下迷宮 | 魔王討伐 | 錬金毒・毒手 | グレー+アンバー+赤 |
| 電脳ダイブ／システム侵入 | 中枢AI（カーネル）へ | ワーム・ロジックボム（感染拡大） | 暗端末+ネオンシアン |
| 深海降下 | 深海の主（"深淵"が文字通り） | 有毒生物・酸 | 深青→黒+ティール |
| 和風・地獄めぐり | 獄の主（閻魔/鬼王）へ | 呪詛・穢れ（祟りの伝播） | 墨+朱+金 |

選んだらUIパレットと系統名の読み替えを同時に設計する。

## §24. 等価性検証戦略

### 24.1 目的

TS移植がPython版と数値的に等価であることを証明し、凍結バランスを完全に引き継ぐ。

### 24.2 二段階の検証

**ステップ1：決定論テスト（完全一致）**

```bash
# Python側：代表シードNで全ターンの状態ハッシュをフィクスチャ出力
python3 -m roguelike export-fixture --seed 7F3A --seed A1B2 > tests/fixtures/

# TS側：同じシードで再生し、各ターンの状態と突き合わせ
npx vitest run tests/golden.test.ts
```

1手でもズレたら検出。RNG分離（マップ用・ドロップ用）も再現すること（§11）。

**ステップ2：分布等価テスト（統計的等価性）**

TS版でBot 1,000ランを回し、KPI分布がPython版と統計的に区別つかないことを確認。

| KPI | Python版（基準） | 検定 | 合格条件 |
|---|---|---|---|
| クリア率 | 58.5% | 比率の差のz検定 | p > 0.05 かつ差が±5%以内 |
| ビルド分布 | thorns31/power28/poison21/sustain16/utility4% | カイ二乗検定 | p > 0.05 |
| 死因分布 | ボス63/射手15/大兵14/その他8% | カイ二乗検定 | p > 0.05 |

### 24.3 CI への追加（dev guide §8 の `.github/workflows/ci.yml` に追記）

```yaml
- name: TS golden tests
  run: npx vitest run tests/golden.test.ts

- name: TS distribution check (n=200, fast)
  run: npx ts-node scripts/sim.ts --n 200 --assert-kpi
```

## §25. 実装ロードマップ（推奨順）

### Step 1：data.json 確定（ブロッカー）
- data.py の ENEMIES / SPAWN_TABLE / WEAPONS / RELICS を `data.json` の null に流し込む
- Python版が data.json を読んで動くことを確認（ロジック無変更）
- Botシム n=300 でKPIが基準値に留まることを確認

### Step 2：TSプロジェクト雛形
- `web/` ディレクトリに Vite + TypeScript 初期化
- data.json から型定義を自動生成（`json-schema-to-typescript` 等）
- data.json ローダー + バリデーション実装

### Step 3：TSエンジン移植 + ゴールデンテスト
- `engine.py` を `engine.ts` に忠実移植（同一ロジック・同一RNG）
- §24.2 ステップ1のゴールデンテストを全シードで通過させる
- §24.2 ステップ2の分布チェックを通過させる
- CI に §24.3 を追加

### Step 4：プレイ画面実装
- HTML/CSS 3領域レイアウト（§21.2）
- グリッド描画 + WASD入力（§22）
- 予兆ハイライト・ダメージプレビュー・ルックモード（§21.3・§22.4）

### Step 5：オーバーレイ実装
- 報酬3択モーダル（カードUI・トレードオフ明示）
- 死亡画面（KPIデータ可視化・クイックリスタート）

### Step 6：テーマ適用（§23決定後）
- CSS変数でカラーパレット差し替え
- data.json のテキストフィールド（name等）を世界観に合わせて更新
- タイルシンボル・ログテキストをテーマの語彙に統一
